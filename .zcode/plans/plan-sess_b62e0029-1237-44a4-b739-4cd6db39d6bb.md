## Admin UI Redesign — Phase 1 (Core Foundation)

**Goal:** Transform the admin from a 0/10 inconsistent ad-hoc UI into a clean, modern (Stripe/Linear/Vercel-style) admin: white/neutral surfaces, ONE accent, soft shadows, rounded corners, generous whitespace, consistent components. Keep it easy to use and handle.

**Stack reality:** SvelteKit + Svelte 5 (runes) + Tailwind v4 (CSS-first `@theme` in `app/src/routes/layout.css`). I will NOT change the storefront, routes, server logic, auth, or data shape — purely UI/components.

### Step 1 — Design tokens (`app/src/routes/layout.css`)
Add a small admin token set to the existing `@theme` block so colors are centralized, not hardcoded per page:
- `--color-admin-primary: #354238` (existing brand green-black — sophisticated, on-brand)
- `--color-admin-primary-hover: #24302a` (existing gray-900)
- Keep existing `gray-*` neutral ramp and `Inter`/`Playfair` fonts.
- Sidebar logo keeps `font-serif` for brand identity; all UI text = Inter (cleaner admin feel).

### Step 2 — Reusable admin UI library (`app/src/lib/components/admin/ui/`)
New components (Svelte 5 runes, snippet-based). This is the heart of the fix — pages stop reinventing primitives:
1. **`PageHeader.svelte`** — title + subtitle + optional back button + `actions` snippet (replaces ~7 ad-hoc headers)
2. **`Card.svelte`** — rounded panel with optional title/description header (`header`/`footer` snippets)
3. **`StatCard.svelte`** — icon swatch + label + value + optional sub-text
4. **`Badge.svelte`** — semantic pill: `neutral | success | warning | danger | info`
5. **`Button.svelte`** — variants `primary | secondary | ghost | danger`; renders `<a>` or `<button>`; optional icon
6. **`Table.svelte`** + `Th`/`Td`/`Tr` helpers — consistent header/row/empty-state styling
7. **`Field.svelte`** — label + control + help text (label-inside-input optional prefix like "Rs.")
8. **`TextInput.svelte` / `Select.svelte` / `Textarea.svelte`** — styled form controls
9. **`Modal.svelte`** — backdrop dialog, focus trap-friendly, `title`/`description` + actions snippet (replaces bespoke modals in order detail, product edit, users)
10. **`EmptyState.svelte`** — icon + title + description
11. **`status.ts`** — single source of truth: order/product/customer status → `{ label, tone }`. Kills the copy-pasted `statusClass`/`statusLabel` functions.
12. **`index.ts`** — barrel export

### Step 3 — Rebuild the shell (`app/src/routes/abayiza-secure-admin-7k9x2p/+layout.svelte`)
- Cleaner sidebar: softer active state (subtle `bg-gray-100` + accent text instead of full black block), grouped nav (Orders group, Catalog group, Customers group, System group) with section labels, collapsible on desktop + slide-over on mobile (keep existing behavior).
- Cleaner header: keep search + user menu + logout; remove non-functional notification dot noise or keep subtle; use new `Button`.
- Toast: keep working flash-toast system, restyle with `Badge`/cleaner card.
- No changes to `+layout.server.ts`, auth, or `hooks.server.ts`.

### Step 4 — Refactor Phase-1 pages onto the new components
These are the highest-traffic pages; all use the shared primitives so they become consistent + shorter:
- **Dashboard** (`+page.svelte`) — `PageHeader`, `StatCard` grid, `Card` for revenue/list/orders/low-stock, `Badge` for status, `EmptyState`.
- **Products list** (`products/+page.svelte`) — `PageHeader`, `Button`, filter tabs, `Table`, `Badge`, cleaner action buttons.
- **Product new** (`products/new/+page.svelte`) — `PageHeader` (back), `Card` sections, `Field`+inputs, reuse existing `ImageFileQueue` (restyled to match), cleaner variant rows.
- **Product edit** (`products/[id]/+page.svelte`) — same as "new" + `Modal` for delete confirm.
- **Orders list** (`orders/+page.svelte`) — `PageHeader`, filter form, `Table`, `Badge` via `status.ts`.
- **Order detail** (`orders/[id]/+page.svelte`) — `PageHeader`/hero, `Card` sections, `Badge`, `Modal` for cancel confirm, `Button` status actions.

customers, users, categories, reviews, analytics, storefront, settings, login — these keep working unchanged; they'll be migrated in Phase 2 using the same component library. I will not touch their server logic or data.

A cohesive, easy-to-use admin shell + the 6 most-used pages, built on a reusable component library that makes the remaining pages fast and consistent to migrate next. No backend/behavior changes — same forms, same actions, same data, just genuinely usable UI.
